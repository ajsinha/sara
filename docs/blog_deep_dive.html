PART I

Knowledge Distillation: Teacher → Student

Knowledge distillation (Hinton et al., 2015) trains a smaller student model to replicate the behaviour of a larger teacher model. The teacher produces "soft targets" — probability distributions over outputs at a raised temperature T — that encode dark knowledge: rich inter-class similarity information that hard one-hot labels throw away.

Figure 1 — Standard KD: the T² factor restores gradient magnitude lost by softening.

Unpacking the Loss — Term by Term

The formula above is the heart of all knowledge distillation. Every term has a specific job:

TermNameWhat It Does
α Mixing coefficient A scalar in [0, 1] that balances the two loss terms. α = 1 means "learn entirely from the teacher." α = 0 means "ignore the teacher, use hard labels only." Typical values: 0.4–0.7. Higher α when the teacher is strong and data is small.
T Temperature Softens the probability distributions. At T=1, softmax is sharp and peaked. At T=4, it flattens, revealing inter-class similarities — the dark knowledge. Both teacher and student logits are divided by T. Typical: 2–8 (vision), 4–16 (NLP).
T² Gradient restoration When you soften by T, the gradients shrink by T². Without this correction, the KL term becomes negligible at high temperatures and the CE term dominates regardless of α. T² keeps both loss components on the same scale.
KL(student ‖ teacher) Distillation term Kullback-Leibler divergence between the student's soft distribution and the teacher's. Measures how different the student's predictions are from the teacher's, including all the nuanced inter-class structure. KL = 0 means identical. This is what transfers the dark knowledge.
CE(student, labels) Task term Standard cross-entropy between the student's predictions (at T=1, not softened) and the ground-truth one-hot labels. Ensures the student learns the correct answers — not just how to mimic the teacher's mistakes.
1 − α Hard-label weight The complement of α. As α increases, more weight goes to soft targets, less to ground truth. The two weights always sum to 1.
Intuition

The first term says "make your output distribution look like the teacher's" — including all the nuanced inter-class similarity (a cat is more like a dog than an airplane). The second term says "but also get the right answer." Temperature controls how much dark knowledge is visible. Alpha controls how much you trust the teacher versus the ground truth.

Why This Matters for LLMs

Large foundation models (GPT-4, Claude, Llama 70B) require tens of gigabytes of VRAM. KD compresses their capability into models that run on laptops and phones. DistilBERT retains 97% of BERT's GLUE performance at 40% fewer parameters. Microsoft's Phi-3 (3.8B) outperforms Llama-2-7B on reasoning — distilled from GPT-4-generated synthetic data.

The API-Only Problem

Standard KD requires gradient access to both models. But in production RAG systems, models are often API-only — you can't access weights, you can't fine-tune, you can't compute KL divergence over logits. All you have is the system prompt and text inputs/outputs.

This is the gap KD-SPAR fills: knowledge distillation applied to the prompt rather than the weights, using the teacher's output distribution as the optimisation signal.

The Self-Knowledge Hypothesis

Central Claim

The student model has privileged self-knowledge about what instructions elicit its best behaviour. Because it will execute the instruction, it knows how different phrasings affect its own outputs — knowledge that an external optimiser doesn't have. Therefore, student self-authored instructions should outperform externally proposed instructions given the same KD scoring signal.

This is testable. The A−B gap — KD score of self-proposed (A) minus externally-proposed (B), with everything else identical — isolates the pure value of self-authorship.

Phase 1 — Diagnostic Pass

The student runs on every training query with the current prompt P. Each response is scored against the cached teacher response using a composite KD score:

Function — KD_score (v1, Jaccard-based)
Input: student response s, teacher response t
Output: alignment score ∈ [0, 1]
t_cited ← has_citation(t)  ▹ regex match for [Doc-N]
s_cited ← has_citation(s)
cit ← if (¬t_cited) ∨ s_cited then 1.0 else 0.0
return 0.3 × cit + 0.7 × jaccard(s, t)

Failures are classified into five modes: missing_citation, over_hedged, under_hedged, incomplete, format_drift. The worst-scoring queries are selected for the self-interview.

Phase 2 — Self-Interview

This is the core innovation. The student is shown its own failure and asked to propose a fix. The prompt template is precisely structured:

Prompt 1 — Self-Interview Template
System Context You are analysing your own performance on a retrieval-augmented generation task. Input Variables QUERY: {query} YOUR RESPONSE: {student_response} FAILURE MODE: {failure_description} TARGET PATTERN: {target_pattern} Task In exactly ONE sentence, propose a specific instruction that should be added to your system prompt to help you avoid this failure in future responses. Rules • Write ONLY the instruction text (no preamble, no explanation, no quotes) • Be specific and actionable ✗ BAD: "be more careful" ✓ GOOD: "Always cite retrieved passages using [Doc-N] notation after each claim" • Focus exclusively on the described failure mode

Phase 3 — Aggregation

Proposals are embedded using MiniLM-L6 sentence embeddings, clustered via agglomerative clustering, and each cluster representative is scored on a mini evaluation set. The top-K proposals by KD score improvement are selected. This deduplicates semantically similar proposals — five variations of "cite your sources" collapse into one representative.

Phase 4 — Validate & Commit

The candidate prompt (current P ⊕ selected instructions) is tested on held-out validation queries that were never used in diagnosis or scoring. If the aggregate KD score improves by at least threshold δ, the update is committed. Otherwise it reverts entirely. This gate prevents the prompt from accumulating bad instructions — a rejected iteration leaves P unchanged.

PART V

Why Base KD-SPAR Failed — A Root Cause Analysis

0 out of 3 SPAR iterations were accepted in most runs. The proposals were either too generic ("Always cite sources") or semantically good but penalised by Jaccard. Three specific root causes:

1. Meta-Cognitive Capacity Gap

The self-interview asks a 3B model to simultaneously: (a) understand what it did wrong, (b) reason about what instruction would change its behaviour, and (c) generate that instruction precisely. This is a meta-cognitive task that requires the model to model itself. At 3B parameters, this capacity is insufficient. The teacher (8B) has 2.7× more parameters for exactly this kind of abstract reasoning — which is why Condition B (teacher proposes) outperforms Condition A (student proposes).

2. Jaccard Penalises Good Paraphrases

The KD score uses 70% Jaccard token overlap. But a student that paraphrases the teacher correctly — using different but equivalent words — gets a lower score than one that copies tokens verbatim. The validate-and-commit gate then rejects the improvement because the metric can't see it. This is a fundamental mismatch between what we're optimising (semantic alignment) and what we're measuring (token overlap).

3. Cold Start + Strict Gate + Few Iterations

Starting from a vanilla prompt with δ ≥ 0.003 and only 3 iterations: the first proposals are necessarily broad (the prompt has no accumulated context). Broad proposals rarely clear a strict threshold. With only 3 chances, the loop never gets past the first "bad proposal" phase into the "targeted refinement" phase that would produce real improvements.

PART VI

Eight Enhancements — Enhanced KD-SPAR

Each root cause got a targeted fix. Together they form Condition F.

01 — HYBRID PROPOSER
Teacher Diagnoses, Student Proposes

Separates the hard part (diagnosis — requires meta-cognition) from the privileged part (proposal — requires self-knowledge). The teacher's 8B capacity handles abstract failure analysis; the student's 3B self-knowledge handles instruction authoring.

02 — BERTSCORE METRIC
Semantic Replaces Syntactic

Score becomes 0.3×citation + 0.5×BERTScore(MiniLM) + 0.2×Jaccard. Paraphrases that capture meaning score high. The gate finally rewards semantic improvement.

03 — CONTRASTIVE INTERVIEW
Good/Bad Pair Reasoning

Show the student a query where it scored well and one where it scored poorly. "What did you do right here that you missed there?" Concrete positive examples produce more specific proposals.

04 — WARM-START FROM B
Bootstrap from External Proposal

Run one teacher-proposed iteration first. The self-interview then diagnoses from an already-improved prompt — not the vanilla default.

05 — MORE ITERATIONS (5)
Room to Converge

First proposals are broad; later ones are targeted. 3 iterations isn't enough. 5 gives the loop time to refine.

06 — SOFT COMMIT GATE
Simulated Annealing

Accept with probability exp(δ/T), temperature decreasing per iteration. Marginal improvements get through and compound over iterations.

07 — TEACHER-GUIDED INTERVIEW
Show the Actual Teacher Response

Include the teacher's full response text in the interview prompt — not just a failure mode label. The student sees what "good" looks like concretely.

08 — TREE OF THOUGHT
Structured Proposal Search

Replace flat single-shot generation with Branch → Evaluate → Expand. Generate K root-cause hypotheses, score each for explanatory power, expand the best into targeted instructions. Explores the reasoning space about why failures occurred.

Prompt 2 — Tree of Thought Branch Template
System Context You are analysing WHY your response failed on this query. Input Variables QUERY: {query} YOUR RESPONSE: {student_response} TEACHER RESPONSE: {teacher_response} FAILURE MODE: {failure_mode} Task Think about the ROOT CAUSE of this failure — not the surface symptom but the underlying reason your response diverged from the teacher's. Generate {K} DIFFERENT hypotheses about WHY you failed. Each hypothesis must suggest a fundamentally different root cause. Output Format HYPOTHESIS 1: {one sentence — root cause A} HYPOTHESIS 2: {one sentence — root cause B} HYPOTHESIS 3: {one sentence — root cause C}

Why this works better than flat generation: A single-shot interview often produces 5 slightly different wordings of "cite your sources." Tree of Thought forces the student to reason about why it failed first — and different root causes ("doesn't know when to cite" vs. "treats citations as optional" vs. "cites only direct quotes") produce fundamentally different instructions. The evaluate step prunes bad reasoning paths before they generate instructions, saving compute and improving quality.

PART VII

What the Data Actually Says

Finding 1: The Self-Knowledge Hypothesis Is Capacity-Gated

At 3B parameters, the student cannot effectively self-diagnose. The A−B gap is negative (−0.025). But with hybrid diagnosis (teacher diagnoses, student proposes), the Enhanced variant outperforms even the teacher's own proposals (F−B = +0.014). The privileged self-knowledge is real; it needs better infrastructure to extract.

Finding 2: MetaKDSPAR Works Without the Other Enhancements

Condition E (4 specialists + conductor, no BERTScore, no warm-start, no ToT) matched the fully Enhanced variant at 0.312. Multi-perspective diagnosis alone — decomposing "what went wrong" into citation, calibration, completeness, and format specialists — catches compound failures that the monolithic classifier misses. E−A gap = +0.040.

Finding 3: The Metric Matters More Than the Algorithm

Jaccard token overlap actively penalises good paraphrases. Switching to BERTScore in the Enhanced variant was likely the single highest-impact change. If your optimisation loop uses the wrong fitness signal, the search algorithm doesn't matter.

Finding 4: The Baseline Ladder Behaves as Expected

B > D (KD-guided proposal adds value over no tuning), C < D (random instructions hurt — they don't just fail to help, they actively degrade). This confirms the experimental design is sound: the KD signal is informative, and adding random instructions is worse than doing nothing.

The Takeaway

KD-SPAR's intellectual bet — that a model has useful self-knowledge about what instructions work — is not wrong, but it's harder to extract than expected. At 3B parameters, you need: (1) the teacher's help for diagnosis, (2) a semantic metric that doesn't penalise paraphrases, (3) structured reasoning (ToT) instead of flat proposal generation, and (4) a warm-start to avoid the cold-start problem. With all eight enhancements, the self-knowledge mechanism outperforms everything — including pure external proposal.

PART IX

Strengths, Limitations, and Future Directions

Strengths

• API-only: Works without weight access — applicable to any LLM provider.
• Fully reproducible: temperature=0.0, fixed seeds, local models, zero API cost.
• Honest: The paper and blog report negative results alongside positive ones. The adaptive commentary system generates honest analysis for every outcome.
• Practical: The Federated variant addresses real enterprise needs (multi-site RAG with data sovereignty). The Enhanced variant works on current hardware with current models.

Limitations

• Metric quality: Even BERTScore is an automatic proxy. Human evaluation (3 raters, Cohen's κ > 0.6) is needed for top-venue submission.
• Single domain: All experiments use RAG QA. Generalisation to code generation, structured extraction, or creative writing is not yet established.
• No convergence guarantee: The loop uses a fixed iteration count. There is no formal proof that the prompt will converge.
• 3B capacity ceiling: The base self-knowledge hypothesis may hold at 7B+ but we haven't tested it.

Future Directions

• Larger students (7B+) — test whether the base A−B gap becomes positive when the student has more meta-cognitive capacity.
• BERTScore throughout — run all conditions on the same semantic metric for fairer comparison.
• Active learning + adversarial SPAR — the student proposes queries it's most uncertain about.
• Learned specialist selection — train a router that predicts which MetaKDSPAR perspectives are most informative per query, reducing overhead from 4× to ~2×.
• Formal convergence analysis — prove conditions under which the SPAR loop converges.
