Distributed Training: Scaling Model Training Across Multiple GPUs

Training large language models requires distributing computation across multiple GPUs and
often multiple machines. The three primary parallelism strategies — data, tensor, and
pipeline — can be combined to train models that exceed the memory capacity of any single
accelerator.

Data Parallelism

Data parallelism replicates the entire model on each GPU and splits the training batch
across them. Each GPU computes gradients on its local mini-batch, then all gradients are
synchronized through an all-reduce operation before the optimizer step. This is the simplest
form of parallelism and scales well to 8-64 GPUs for models that fit in a single GPU's
memory. The main overhead is the all-reduce communication, which transfers approximately
2 * model_size bytes per step. For a 7B FP16 model, this is approximately 28 GB per step,
which takes 1-3 seconds over 100 Gbps InfiniBand.

The effective batch size equals per-GPU batch size times the number of GPUs. Large effective
batch sizes can destabilise training — learning rate warmup and linear scaling (multiply
learning rate by the parallelism factor) are standard mitigations. Gradient accumulation
allows simulating larger batch sizes without increasing memory, by accumulating gradients
over multiple forward-backward passes before applying the optimizer step.

DeepSpeed ZeRO (Zero Redundancy Optimizer)

Standard data parallelism wastes memory because every GPU stores a full copy of the model
weights, optimizer states, and gradients. DeepSpeed ZeRO eliminates this redundancy through
progressive sharding across three stages.

ZeRO Stage 1 partitions optimizer states across GPUs. For Adam, this reduces per-GPU memory
by approximately 4x (Adam stores momentum and variance for each parameter). ZeRO Stage 2
also partitions gradients, reducing memory by an additional 2x. ZeRO Stage 3 partitions
the model parameters themselves — each GPU only stores 1/N of the parameters and gathers
them on-demand for computation. Stage 3 enables training models much larger than single-GPU
memory but introduces significant communication overhead.

FSDP (Fully Sharded Data Parallel) is PyTorch's native implementation of ZeRO-3 concepts.
It shards parameters, gradients, and optimizer states across data-parallel ranks and uses
all-gather operations to reconstruct full parameters for each layer's forward and backward
pass. FSDP is now the standard for training 7B-70B models on multi-GPU setups.

Tensor Parallelism

Tensor parallelism splits individual layers across GPUs. For a linear layer W with output
dimension d, tensor parallelism splits W into N column-wise shards, each processed by one
GPU. The partial outputs are combined through an all-reduce operation. Megatron-LM
implements tensor parallelism for transformer attention and FFN layers, enabling models
whose individual layers exceed single-GPU memory. Tensor parallelism is most effective
within a single node (using fast NVLink interconnects) because it requires frequent
communication at every layer.

Pipeline Parallelism

Pipeline parallelism assigns sequential blocks of layers to different GPUs. GPU 0 runs
layers 1-8, GPU 1 runs layers 9-16, and so on. The main challenge is the pipeline bubble —
while GPU 0 is computing the forward pass for micro-batch 1, GPUs 1-N are idle. Micro-
batching (splitting the batch into smaller micro-batches that flow through the pipeline)
reduces the bubble, and interleaved scheduling (1F1B — one forward, one backward per
micro-batch) further improves utilization.

Mixed Precision Training

Mixed precision training uses FP16 or BF16 for forward and backward passes while maintaining
FP32 master weights for the optimizer step. This halves the activation memory and doubles
throughput on GPUs with tensor cores (all NVIDIA GPUs from Volta onwards). BF16 is preferred
over FP16 for training because its larger exponent range avoids the overflow and underflow
issues that require loss scaling with FP16. Dynamic loss scaling automatically adjusts the
scaling factor to prevent gradient underflow in FP16 training.

Gradient Checkpointing

Gradient checkpointing (activation recomputation) trades compute for memory by discarding
intermediate activations during the forward pass and recomputing them during the backward
pass. This reduces activation memory from O(N * L) to O(sqrt(N * L)) where N is sequence
length and L is the number of layers. The compute overhead is approximately 33% (one
additional forward pass per checkpointed segment). Selective checkpointing only recomputes
the most memory-intensive operations (attention scores) while keeping cheaper activations.

Communication Optimization

NCCL (NVIDIA Collective Communications Library) provides optimized implementations of
all-reduce, all-gather, reduce-scatter, and broadcast operations across GPU topologies.
NVLink provides 600 GB/s bandwidth between GPUs within a node (A100) while InfiniBand
provides 200-400 Gb/s between nodes. The 3:1 bandwidth ratio between intra-node and
inter-node communication motivates placing tensor parallelism within nodes and data/pipeline
parallelism across nodes. Typical training clusters use 8 GPUs per node with 4-way tensor
parallelism and 2-way data parallelism within a node.
